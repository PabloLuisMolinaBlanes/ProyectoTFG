se utiliza para informar sobre tendencias centrales de un conjunto de datos. Es vulnerable a valores extremos, haciéndola una métrica no robusta.Para un conjunto de datos $X = \{x_i\,\mid\, i \in S \subseteq \N \}$, se define la media aritmética como:\[{m}=\frac{1}{n}\left (\sum_{n = 1}^{n}x_i\right)=\frac{x_1+x_2+\dots+x_n}{n}\]\subsubsection{Porcentajes}\subsubsection{Detección de colisiones}\subsection{Técnicas y métricas cognitivo-psicológicas}\subsubsection{Tiempo de reacción}\subsubsection{Tiempo de anticipación}\subsubsection{Coordinación mano-ojo}\section{Legalidad y conformidad a la legislación Española y Europeas}\subsection{Protección de Datos}\subsection{Legislación referida a Dispositivos Médicos}\subsection{Propiedad intelectual}\subsubsection{Licencias}\section{Metodología de desarrollo}La metodología de desarrollo principal que se ha utilizado para el proyecto ha sido la metodología Agile.Bajo la metodología Agile, se implementa el software a lo largo de una serie de sprints; Es generalmente utilizado en el desarrollo de prototipos, y contrasta con los circuitos soldados o circuitos impresos.\cite{eswiki:172543964}\cite{breadboard1}\subsubsection{Botón pulsador electrónico}Un button pulsador electrónico consiste en una resistencia que no deja pasar la corriente cuando se activa un pulsador.\cite{pushbuttonswitch}\subsubsection{Potenciómetro}Un potenciometro consiste en una resistencia que se puede configurar con un mango. Un potenciómetro puede cambiar su resistencia con respecto al ángulo de rotación de forma linear o logarítmica.\cite{potentiometer1}\subsubsection{Cableado USB (Universal Serial Bus)}Se utiliza un cable USB en este proyecto para realizar comunicación entre el dispositivo Raspberry Pi 5 y el dispositivo ESP-32 (se explicarán estos en la siguiente sección. Este cable puede realizar funciones útiles en el desarrollo de sistemas empotrados; como cableado serial y de posibilidad de subir firmware a los microcontroladores. Es un cable muy versatil y de gran utilidad en el desarrollo de sistemas embebidos.El cable específico que se utiliza para el proyecto es un cable USB-C a USB.\subsection{Microcontroladores}Se ha utilizado un microcontrolador ESP-32 para el proyecto.\subsection{Ordenadores monoplaca}Se ha utilizado un dispositivo Raspberry PI 5 para realizar las funciones de computación general dentro del proyecto.\section{Diseño software}\subsection{Sistemas empotrados}\subsubsection{Arquitectura software}La arquitectura que se ha utilizado para el funcionamiento del software localizado en el sistema empotrado es una arquitectura \textbf{basada en módulos}.Es una arquitectura principalmente basada en polling, para recibir y actuar con respecto a eventos, ya sea su proveniencia desde el cable USB-to-UART conectado al dispositivo, o de los dispositivos electrónicos a los que se encuentra conectado el microcontrolador.No solo se reciben eventos; que interpreta como 0, o como 1.  \item Si el programa lee que el voltaje obtenido no es 1 (es decir, el usuario ha accionado el botón y por tanto, no lee su propia corriente la placa), entonces, envía a través del cable serial un valor arbitrario, indicando que el botón se ha accionado.  \item Por último, duerme 100 milisegundos antes de realizar un nuevo ciclo.\end{enumerate}\textbf{Segunda examinación}Esta parte se implementa dentro del fichero denominado secondEste programa sigue un bucle parecido al de la primera examinación; aparecerá en pantalla dos botones; uno para la primera examinación, y el segundo para la segunda examinación.Al pinchar el botón, el usuario debe accionar los dispositivos electrónicos correspondientes a cada examinación: el botón para el primer examen psicotécnico, y las manivelas para el segundo examen psicotécnico.Tras terminar la examinación, el programa se cerrará automáticamente, y almacenará los datos donde se encuentre el programa de examinación correspondiente, en un archivo con formato .txt\section{Subir datos a la nube}Para subir los datos obtenidos de una examinación a la nube; se deberá abrir el programa CLI con el comando \textit{npm run dev}, tras realizar esto; se deberá accionar la segunda opción.El usuario escribirá su nombre legal completo, sus datos de documentación personal, y todo lo demás; el doctor aquí deberá presentar al examinado el formulario de protección de datos correspondiente a sus datos; el cual deberá firmar antes que sus datos se puedan subir a la nube.Tras hacer todo esto, se confirman los datos mostrados en pantalla, y se suben a la nube.\section{Obtener datos de la nube}Para obtener datos de la nube, basta con acceder al programa CLI, accionar la acción número 2, y se creará una tabla dentro de un archivo .html con los datos personales de los examinados; incluyéndose sus resultados en cada una de las pruebas de examinación.\umaappendix{Snippets de código interesantes}\section{Esquema de datos y bases de datos}Los datos se dividen en dos bases de datos.\subsection{profile}Esta base de datos incluye los datos relativos a hospitales;
